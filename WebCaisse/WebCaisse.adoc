=== Question 1.2 — Pourquoi l’envoi du mot de passe par mail est une faille critique + processus sécurisé

Envoyer un mot de passe par mail est bien évidemment une faille critique car le mail peut être intercepté, piraté ou lu par une personne non autorisée.

Cela menace plusieurs élements du dicp :

* la *confidentialité* : le mot de passe est exposé ;
* l’*intégrité* : un tiers peut se connecter et modifier les données ;
* la *disponibilité* : le vrai client peut perdre l’accès à son compte ;
* la *preuve* : on ne peut pas garantir que seul le bon utilisateur a reçu et utilisé le mot de passe.

Au niveau du *RGPD*, ce n’est pas conforme car un mot de passe est une donnée sensible d’authentification. Il ne doit pas être transmis en clair, or dans le cas présent, il l'est.

==== Processus sécurisé proposé

. Le client remplit le formulaire d’inscription.
. Le système enregistre le compte avec un mot de passe *haché* (très important).
. Le système envoie un mail contenant un *lien d’activation temporaire* avec un jeton unique.
. Le client clique sur le lien.
. Il choisit lui-même son mot de passe.
. Le mot de passe est haché avant stockage.
. Le jeton expire après un délai court et ne peut être utilisé qu’une seule fois.

=== Question 1.3 — Pourquoi la table proposée ne permet pas l’historique

tout d'abord, vérifions la table proposée : `FormuleSouscrite(idPointDeVente, idFormule)`

Cette structure ne permet pas l’historique car elle ne contient aucune date.

On ne sait donc pas :

* quand une formule a commencé ;
* quand elle s’est terminée ;
* combien de temps elle a été active ;
* combien de changements ont eu lieu dans le mois.

Donc on ne peut pas faire une facturation mensuelle correcte.

=== Question 1.4 — Correction de la structure

voici donc la table que je propose :

`FormuleSouscrite`

* `idSouscription`
* `idPointDeVente`
* `idFormule`
* `dateDebut`
* `dateFin`

Clé primaire : `idSouscription`

Clés étrangères :

* `idPointDeVente`
* `idFormule`

Cette structure nous permet ainsi de garder toutes les périodes d’activation de chaque formule.

maintenant, nous pouvons nous attaquer au vif du sujet :

[source,sql]
----
DELIMITER $$

CREATE TRIGGER tg_audit_changement_formule
AFTER UPDATE ON FormuleSouscrite
FOR EACH ROW
BEGIN
    IF OLD.idFormule <> NEW.idFormule THEN
        INSERT INTO AuditChangementFormule
        (idPointDeVente, ancienneFormule, nouvelleFormule, dateChangement)
        VALUES
        (OLD.idPointDeVente, OLD.idFormule, NEW.idFormule, NOW());
    END IF;
END$$

DELIMITER ;

=== Question 2.3 — Requête SQL + code Java sécurisé avec `PreparedStatement`

==== Requête SQL

[source,sql]

SELECT c.nom, c.prenom, c.tel, c.mail
FROM Conso c
JOIN Vente v ON v.idConso = c.id
WHERE c.id NOT IN (SELECT id FROM ConsoFidele)
AND v.dateVente BETWEEN ? AND ?

----

==== Code Java

[source,java]
----
public static ArrayList<Conso> listeConsoAFideliser(int seuilVentes, Date dateDeb, Date dateFin) throws SQLException {
    ArrayList<Conso> lesConsos = new ArrayList<>();

    Connection dbConnect = DriverManager.getConnection("jdbc:mysql:" + dbURL, adminUser, adminPwd);

    String requete = "SELECT c.nom, c.prenom, c.tel, c.mail " +
                     "FROM Conso c " +
                     "JOIN Vente v ON v.idConso = c.id " +
                     "WHERE c.id NOT IN (SELECT id FROM ConsoFidele) " +
                     "AND v.dateVente BETWEEN ? AND ? " +
                     "GROUP BY c.id, c.nom, c.prenom, c.tel, c.mail " +
                     "HAVING COUNT(v.id) > ?";

    PreparedStatement pstmt = dbConnect.prepareStatement(requete);
    pstmt.setDate(1, new java.sql.Date(dateDeb.getTime()));
    pstmt.setDate(2, new java.sql.Date(dateFin.getTime()));
    pstmt.setInt(3, seuilVentes);

    ResultSet res = pstmt.executeQuery();

    while (res.next()) {
        Conso leConso = new Conso(
            res.getString("nom"),
            res.getString("prenom"),
            res.getString("mail"),
            res.getString("tel")
        );
        lesConsos.add(leConso);
    }

    res.close();
    pstmt.close();
    dbConnect.close();

    return lesConsos;
}
----

public static double compareLieuVente(ArrayList<ConsoFidele> lesConsos)  {
    double totalEcom = 0;
    double totalMag = 0;

    for (ConsoFidele cf : lesConsos) {
        for (Vente uneVente : cf.getVentes()) {
            if (uneVente instanceof VenteEcommerce) {
                totalEcom += uneVente.getMontantVente();
            } else if (uneVente instanceof VenteMagasin) {
                totalMag += uneVente.getMontantVente();
            }
        }
    }
    return totalMag / totalEcom;
}

=== Question 3.5 — Pourquoi `return totalMag/totalEcom;` est dangereux + correction

C’est très dangereux car si `totalEcom` vaut `0`, alors on fera une division par zéro.

voici la gorrection que je propose :

[source,java]
----
if (totalEcom == 0) {
    return 0;
}
return totalMag / totalEcom;
----
