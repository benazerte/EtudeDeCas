== Mission 1 - Gestion des demandes d’insémination

=== Question 1.1

La requête montre que Coopain choisit une paillette :

* du taureau demandé ;
* du type de paillette demandé ;
* avec un stock restant au moins égal à 1 ;
* en prenant le lot le plus ancien disponible, car on cherche la date minimale.

Donc la règle de gestion est : *pour une demande d’insémination, on prend une paillette disponible du bon taureau et du bon type, en privilégiant le lot le plus ancien encore en stock.*

=== Question 1.3

Requête pour obtenir, pour le taureau `FR0103015562`, les tarifs des types de paillette :

[source,sql]
----
SELECT tp.libelle, t.prixPaillette
FROM Tarif t
JOIN TypePaillette tp ON t.idTypePaillette = tp.id
WHERE t.idNationalTaureau = 'FR0103015562';
----

=== Question 1.4

Requête pour obtenir la liste des taureaux avec le stock total restant, triée par stock croissant :

[source,sql]
----
SELECT ta.idNational, ta.nom, SUM(l.stockRestant) AS stockTotalRestant
FROM Taureau ta
JOIN Lot l ON ta.idNational = l.idNationalTaureau
GROUP BY ta.idNational, ta.nom
ORDER BY stockTotalRestant ASC;
----

=== Question 2.3

En cas de perte ou de vol du smartphone, il y a des risques importants pour la base SQLite locale.

==== Confidentialité

Une personne malveillante pourrait accéder :

* aux informations sur les tournées ;
* aux identités des inséminateurs ;
* aux véhicules ;
* aux données liées à l’activité de Coopain.

==== Intégrité

Une personne pourrait :

* modifier des kilomètres ;
* supprimer des tournées ;
* ajouter de fausses données ;
* fausser les remontées vers le serveur central.

==== Solutions de protection

On peut proposer :

* le chiffrement des données du téléphone ;
* un code PIN fort ou une authentification biométrique ;
* une solution MDM pour gérer les mobiles à distance ;
* la possibilité d’effacement à distance ;
* ne pas stocker plus de données que nécessaire en local.

=== Question 2.4

Le stockage ou la transmission du mot de passe en clair est une faille critique.

Pourquoi ?

* si la base est compromise, les mots de passe sont lisibles directement ;
* si le réseau est intercepté, le mot de passe peut être volé ;
* un utilisateur réutilise souvent le même mot de passe ailleurs.

Donc c’est dangereux pour la sécurité des comptes.

==== Recommandations

Il faut :

* ne jamais stocker le mot de passe en clair ;
* stocker seulement un hachage du mot de passe ;
* utiliser une connexion sécurisée en HTTPS/TLS ;
* mettre en place une authentification sécurisée côté serveur.

=== Question 3.3

Une attaque possible est une attaque de type *Man-in-the-Middle*.

Un attaquant placé entre le smartphone et le serveur peut :

* intercepter les données échangées ;
* lire les informations transmises ;
* modifier les données avant qu’elles arrivent au serveur.

Pour sécuriser les échanges, il faut imposer le protocole *HTTPS* avec *TLS* côté serveur.
